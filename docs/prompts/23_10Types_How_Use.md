## 1\. 🔍 프롬프트 유형과 선택 이유

| 구분 | 선택한 유형 | 선택 이유 |
| :--- | :--- | :--- |
| **프롬프트 유형** | **구조적 출력 요청 (Structured Output)** & **역할 기반 분석 (Role-Based Analysis)** 결합 | 모호한 아이디어를 개발/디자인팀이 즉시 활용할 수 있도록 **체계적인 명세서 형태**로 강제하고, PM의 역할을 부여하여 **요구사항 도출**을 실행합니다. |
| **선택 이유** | **구조적 출력 요청:** 요구사항을 기능적/비기능적/데이터/페이지 구조 등 **명확한 카테고리**로 나누어 출력하도록 지시하여, 개발팀이 원하는 **DB 설계와 프론트엔드 구조**를 잡을 수 있는 기반을 제공합니다. |
| | **역할 기반 분석:** 주니어 PM 페르소나를 활용하여, 개발 리더의 피드백을 반영한 **기술적이고 구체적인 언어**로 아이디어를 재정의하도록 유도합니다. |

## 2\. 📝 작성 위해 추가 필요 요건

AI가 추상적인 아이디어를 구체화된 요구사항으로 변환하기 위해, 현재 PM이 가지고 있는 **추상적인 아이디어**를 구체적인 항목으로 정의하고, 이를 **기술적 요구사항**으로 전환하는 과정이 필요합니다.

  * **현재의 추상적 아이디어 (PM):**
    1.  사용자가 식단을 입력하면 칼로리 및 영양소 계산 (핵심 기능).
    2.  깔끔하고 직관적인 대시보드 (UX/UI).
    3.  회원가입 및 로그인 (기본 기능).
    4.  나의 목표 설정 (체중, 칼로리 등).
  * **기술적 변환 목표:** 이 아이디어를 \*\*기능 명세(Feature Spec), 데이터 모델링(DB), 비기능 요구사항(NFR)\*\*의 세 가지 핵심 영역으로 분해하여 도출해야 합니다.

**[요구되는 출력 구조]**
결과를 다음의 **네 가지 필수 섹션**으로 나누어, 각 항목은 번호가 매겨진 리스트 형태로 작성하십시오.

### 1. 기능적 요구사항 (Functional Requirements)

## 4\. 🚀 프롬프트 실행 결과

> 주니어 PM이 개발팀 및 디자인팀에 전달할 수 있는 상세 요구사항 명세서 초안입니다.

### 1\. 기능적 요구사항 (Functional Requirements)

1.  **회원 인증 기능:**
      * FR-1.1: 이메일 기반 회원가입 및 로그인 기능을 제공해야 한다.
      * FR-1.2: 비밀번호 찾기 (이메일 인증) 기능을 제공해야 한다.
2.  **개인 목표 설정:**
      * FR-2.1: 사용자는 초기 가입 시 목표 (체중, 활동량, 주간 목표 칼로리 등)를 입력해야 한다.
      * FR-2.2: 사용자는 '마이페이지'에서 언제든지 목표 설정을 수정할 수 있어야 한다.
3.  **식단 기록 (핵심 기능):**
      * FR-3.1: 사용자는 날짜별 식단 (아침/점심/저녁/간식)을 입력할 수 있어야 한다.
      * FR-3.2: 식단 입력 시, 텍스트 검색을 통해 식품 데이터를 등록할 수 있어야 한다.
      * FR-3.3: 입력된 식단에 대해 **자동으로 총 칼로리, 탄수화물, 단백질, 지방**을 계산하여 보여주어야 한다.
4.  **피드백 및 리포트:**
      * FR-4.1: 대시보드에서 일일 칼로리 목표 달성률을 시각화하여 보여주어야 한다.
      * FR-4.2: 주간/월간 단위로 영양소 섭취 추이 및 목표 달성 그래프 리포트를 제공해야 한다.

### 2\. 데이터 요구사항 및 DB 모델링 요소 (Data & DB Requirements)

1.  **User 엔티티:** `user_id`, `email`, `password_hash`, `target_weight`, `target_calories`, `activity_level`.
2.  **Food\_Item 엔티티 (기초 식품 DB):** `food_id`, `name`, `calories`, `carbs`, `protein`, `fat`, `unit_size` (ex: 100g).
3.  **Meal\_Entry 엔티티 (사용자 식단 기록):** `entry_id`, `user_id (FK)`, `date`, `meal_type` (A/점/저/간식), `food_id (FK)`, `quantity`.
4.  **데이터 무결성:** 모든 필수 입력 필드는 **NULL 값이 허용되지 않도록** DB 설계를 진행해야 한다.
5.  **성능 요구:** 식품 검색 시 **밀리초 단위**의 빠른 응답 속도를 보장할 수 있는 인덱싱 전략을 수립해야 한다.

### 3\. 비기능적 요구사항 (Non-Functional Requirements)

1.  **보안:**
      * NFR-1.1: 사용자 비밀번호는 \*\*단방향 암호화 (Salted Hashing)\*\*하여 저장해야 한다.
      * NFR-1.2: 모든 통신은 **HTTPS**를 통해 암호화되어야 한다.
2.  **성능:**
      * NFR-2.1: 핵심 페이지 (대시보드, 식단 입력)의 로딩 속도는 **3초 이내**여야 한다.
      * NFR-2.2: 일일 식단 기록은 **최대 100만 명** 사용자를 가정하여 확장 가능하도록 시스템을 설계해야 한다.
3.  **사용성:**
      * NFR-3.1: 웹 서비스는 Chrome, Safari, Firefox 등 주요 브라우저에서 동일하게 작동해야 한다.

### 4\. 필수 페이지 구조 및 UX/UI 요소

1.  **로그인/회원가입 페이지:** 명확한 Call-to-Action 버튼, 깔끔한 입력 폼.
2.  **대시보드 페이지 (MVP 메인):**
      * **UI:** 상단에 일일 목표 칼로리 대비 **현재 섭취량**을 표시하는 도넛 차트 (직관성).
      * **UI:** 주간 영양소 (탄/단/지) 달성률을 막대 그래프로 요약 표시.
      * **기능:** '식단 기록하기' 버튼을 가장 눈에 띄게 배치.
3.  **식단 기록 페이지:**
      * **UI:** 검색 창 아래에 검색 결과 리스트 표시.
      * **UI:** 입력 완료된 식단 항목은 카드 형태로 하단에 누적 표시 및 총 칼로리 실시간 업데이트.
4.  **마이페이지/설정 페이지:** 사용자 정보 및 목표 수정 기능 제공.
